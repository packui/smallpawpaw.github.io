  • 多线程 - Object类的notify方法详解

    2018-09-23 /
    • { 多线程 }

    一、notify()方法

    线程1调用共享对象A的 notify()方法后，会唤醒一个在共享变量A上调用 wait()系列方法后被挂起的线程。
    共享变量A上可能会有多个线程在等待，具体唤醒哪个等待的线程是随机的。

    此外，被唤醒的线程不能马上从 wait()方法返回并继续执行，它必须在获取了共享对象A的监视器锁后才可以返回，也就是唤醒它的线程释放了共享变量A上的监视器锁后，被唤醒的线程不一定会获取到共享对象的监视器锁， 因为该线程还需要和其他线程一起竞争该锁，只有该线程竞争到了共享变量A的监视器锁后才可以继续执行。

    类似 wait()系列方法，只有当前线程获取到了共享变量的监视器锁后，才可以调用共享变量的 notify()方法， 否则会抛出IllegalMonitorStateException异常

    二、norifyAll()方法

    结果分析：
    线程 1 和线程2被挂起后，线程3调用 notifyAll()方法会唤醒A的阻塞集合里面的所有线程
    线程 1 和线程2都会被唤醒， 本次线程2先获取到了A上的锁，然后从 wait()方法返回
    线程 2执行完毕后， 线程 1又获取了A上的锁，然后从 wait()方法返回

    需要注意的地方:
    调用共享变量的notifyAll()方法，只会唤醒在调用这个方法前调用了共享变量的wait()系列函数，然后被放入阻塞集合里面的线程。
    如果调用 notifyAll()方法后有线程调用了该共享变量的 wait()方法被放入阻塞集合， 则该线程是不会被唤醒的。

    调用 wait()方法的线程必须事先获取共享对象的监视器锁，否则调用 wait()方法时线程会抛出 IllegalMonitorStateException异常

    (2) 调用该共享变量的方法，并且该方法使用了synchronized修饰:

    一个线程可以从挂起状态变为可以运行状态(也就是被唤醒)， 即使该线程没有被其他线程调用 notify()、 notifyAll()方法进行通知，也可以进行虚假唤醒，方式有：

    在其它线程里被中断，或者等待超时

    虽然虚假唤醒在应用实践中很少发生，但要防患于未然，做法就是不停地去测试该线程被唤醒的条件是否满足，不满足则继续等待，也就是说在一个循环里来进行 wait()方法调用，从而进行防范，退出循环的条件是满足了唤醒该线程的条件

    当线程1调用共享对象A的 wait()方法时，线程1只会释放共享对象A的锁，线程1持有的其他共享对象的监视器锁并不会被释放

    当线程1调用共享变量A的 wait()方法时， 线程1会被阻塞挂起， 直到发生下面几件事情之一才返回：

    (1) 其他线程调用了共享对象A的 notify()或 notifyAll()方法；

    (2) 其他线程调用了线程1的 interrupt()方法， 线程1抛出 InterruptedException异常返回

    二、wait()方法 - InterruptedException异常

    当线程1调用共享对象obj的 wait()方法被阻塞挂起后，如果其他线程中断了线程1

    则线程1会抛出 InterruptedException异常并返回

    三、wait(long timeout)方法：单位毫秒

    该方法相比 wait()方法多了一个超时参数，它的不同之处在于，一个线程1调用共享对象A的wait(long timeout)方

    法被挂起后，如果在指定的 timeout时间内没有其他线程调用A的 notify()或者A的notifyAll()方法进行唤醒，

    那么该方法还是会因为超时而返回。

    如果将 timeout设置为 0则和 wait()方法效果一样，因为在 wait()方法内部其实就是调用了 wait(O)方法

    如果将 timeout设置为负数，则会抛出 IllegalArgumentException异常

    四、wait(long timeout,int nanos)方法

    底层调用wait(long timeout)方法实现

    这个方法类似于wait(long timeout)方法,但是它允许在释放锁之前对时间量进行更精细的控制来等待通知
    以纳秒为单位测量，通过以下方式给出：1000000*timeout+nanos
    在其他所有方面，这个方法与wait(long timeout)方法做的事情一样

    特别的：wait(0,0)与wait(0)的意思相同

    线程会释放此监视器的所有权并等待，直到以下两种情况已经发生：
    (1) 另一个线程通知等待该对象监视器的线程: 通过调用notify()方法或notifyAll()方法唤醒
    (2) 超时，超过timeout指定的毫秒时间加上nanos指定的纳秒时间

    线程会一直等待，直到它可以重新获得监视器锁并恢复执行

  • 多线程 - 进程与线程

    2018-09-05 /
    • { 多线程 }

    进程与线程概念

    进程： 进程是系统进行资源分配和调度的基本单位，操作系统在分配资源时是分配给进程的，一个进程至少包含一个线程，进程中的多个线程可以共享进程资源，通常我们启动一个应用程序就启动了进程，可以看作是应用级别的概念。
    例如：我们启动电脑管家应用程序就会在后台启动一个进程。

    线程： 线程本身不能独立存在，它是进程中的一个执行路径，也就是进程中的一个实体，操作系统分配资源时有个特殊的资源CPU是分配给线程的，因为真正占用CPU的是线程，线程之间在轮流占用CPU，所以，线程又是CPU分配的基本单位，可以看作是功能级别的概念。
    例如：我们使用电脑管家的垃圾清理功能，同时又使用电脑管家的杀毒功能，那么其实就相应的启动了垃圾清理的线程、杀毒的线程。

    进程与线程的内存结构

    堆： 进程中最大的一块内存，在进程创建时分配的区域，存放着new构建的对象实例，可以被进程中所有线程共享。

    方法区： 存放着Java虚拟机加载的类、常量、 静态变量等信息，也可以被进程中所有线程共享。

    程序计数器： 即PC计数器，是线程中的一小块内存区域，用来记录线程当前要执行的指令的地址，是线程私有的，因为线程是CPU分配的基本单位，线程之间在轮流占用CPU，每当本次CPU占用的时间片用完之后，当前线程就要让出CPU的执行权，那么就在这个时候，程序计数器就要记录让出CPU时当前线程执行到哪了，等到下一次线程被重新分配到CPU时，接着这个地方继续向下执行。
    备注：如果执行的是native方法，那么程序计数器记录的是undefined的地址；如果执行的是java代码，程序计数器记录的便是要执行的指令地址。

    栈： 每个线程都有自己的栈区域，它存放着当前线程的局部变量、线程的调用栈帧，也是线程私有的。

    启动线程的正确打开方式是调用start方法，而且调用start方法，线程并不会立即启动，这只是一个启动的标志，就像在告诉系统我已经万事俱备了，就差你给我CPU的执行机会！我就可以真正启动了！
